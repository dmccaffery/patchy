# patchy-config Helm chart

The pipeline switch-on for a [patchy](../patchy) release: the `Integration` and `Forge` custom resources, rendered from
the `integrations` / `forges` values arrays into the release namespace. The controllers idle until at least one of each
exists — an Integration says where findings come from and where the tracking issues go; a Forge says how repositories
are cloned and pushed.

## Why a separate chart

Helm validates every manifest in a release against the API server **before** applying anything. A single chart carrying
both the `patchy.bitwisemedia.uk` CRDs and custom resources of those CRDs therefore fails its first install: the CRs are
validated before the CRDs exist. Splitting the CRs out makes the ordering explicit:

```sh
# 1. the stack: CRDs, controllers, agent sandbox
helm install patchy oci://ghcr.io/bitwise-media-group/patchy/charts/patchy \
    --version <X.Y.Z> --namespace patchy --create-namespace

# 2. the switch-on: Integration + Forge CRs, SAME namespace
helm install patchy-config oci://ghcr.io/bitwise-media-group/patchy/charts/patchy-config \
    --version <X.Y.Z> --namespace patchy -f values.yaml
```

Both charts version 1:1 with the app release. Install this chart into the **same namespace** as the patchy release — the
controllers only reconcile CRs they can see.

## Values

```yaml
integrations:
  - name: github
    spec:
      provider: github
      secretRef:
        name: patchy-github
      interval: 10m
      github:
        issues:
          enabled: true
          approveComment: /approve
        codeScanningAlerts:
          enabled: true

forges:
  - name: github
    spec:
      provider: github
      secretRef:
        name: patchy-github
      interval: 10m
```

Each entry's `spec` is rendered verbatim and validated twice: client-side by `values.schema.json` — which embeds the CRD
spec schemas, regenerated by `mise run codegen` whenever the API types change, so a typo'd or mistyped field fails
`helm install` before anything is applied — and server-side by the CRDs themselves (including the CEL rules the values
schema cannot carry). See [`deploy/kustomize/base/crs.example.yaml`](../../deploy/kustomize/base/crs.example.yaml) for
the full field walkthrough (GHES base URLs, org allowlists, repository regexes). `commonLabels` / `commonAnnotations`
stamp extras on every rendered CR.

The referenced Secrets are created out of band and never owned by either chart — see
[`deploy/kustomize/base/secrets.example.yaml`](../../deploy/kustomize/base/secrets.example.yaml): `appID` + `privateKey`
(or `token`), plus `webhookSecret` for Integrations. One GitHub Secret may serve both CRs, or you can split read and
write identities across two GitHub Apps.

Prefer applying the CRs yourself? Skip this chart and `kubectl apply` the same objects — the chart is a convenience, not
a requirement.

## Publishing

`charts/patchy-config` is packaged and pushed to `oci://ghcr.io/bitwise-media-group/patchy/charts` by
[`.github/workflows/helm.yaml`](../../.github/workflows/helm.yaml) alongside the patchy chart when a release is
published; release-please stamps `version`/`appVersion` in `Chart.yaml` as part of the release PR. Lint locally with
`mise run helm-lint`.
